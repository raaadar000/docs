# Troubleshooting: базовые проблемы

## 1) Не удаётся войти через Google

### Симптом

Кнопка входа есть, но авторизация не завершается.

### Что проверить

1. Вы входите корпоративным аккаунтом (`@ulta.team`).
2. Нет блокировки pop-up/redirect в браузере.
3. Проблема воспроизводится в режиме инкогнито.

### Эскалация

Если вход не работает у нескольких пользователей, эскалируйте разработчику/админу как потенциальную OAuth-проблему.

## 2) Нет проекта в списке

### Симптом

После входа проект отсутствует.

### Что проверить

1. Название проекта сверено с ожидаемым.
2. Проект уже создан в админке.
3. Вы добавлены в деплоеры именно для этого проекта.

### Решение

Запросите у администратора добавление в список деплоеров проекта.

## 3) Нет доступа к `/admin` или не видно админ-функций

### Важно

Это ожидаемое поведение для большинства пользователей.

### Причина

Доступ к админке разрешён только для ограниченного списка лиц (`ADMIN_EMAILS`).

### Что делать

1. Не пытайтесь обходить ограничение.
2. Создайте запрос на действие администратору (или на временное расширение доступа по процессу команды).

## 4) Деплой завершился ошибкой

### Симптом

Статус деплоя неуспешный.

### Базовые шаги

1. Повторите попытку один раз.
2. Убедитесь, что проект корректно настроен в админке (Framer URL, GitHub repo, target folder).
3. Передайте администратору точное время ошибки и проект.

### Для разработчиков

Проверьте server logs и корректность токена GitHub проекта.

## 5) CMS-контент не отображается после экспорта

### Симптом

Страница открывается, но коллекция пустая или появляются runtime-ошибки.

### Что делать

1. Повторно выполните экспорт/деплой.
2. Проверяйте консоль на ошибки вида `ReferenceError: It is not defined`.
3. Если ошибка повторяется, передайте лог разработчику и укажите проблемный URL страницы.

### Примечание

В проекте уже есть исправление для частого кейса range-loader CMS, но новые варианты минификации могут требовать дополнительной проверки.
